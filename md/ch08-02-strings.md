## 使用字符串存储UTF-8编码的文本

我们在第4章中讨论过字符串，但现在我们将更深入地探讨它们。  
新的Rust开发者常常在字符串方面遇到困难，这主要有三个原因：Rust容易引入潜在错误；字符串是一种比许多程序员所认为的更复杂的数据结构；还有UTF-8编码的问题。这些因素结合在一起，可能会让那些来自其他编程语言的人感到难以理解。

我们在集合的上下文中讨论字符串，因为字符串实际上是由字节组成的集合，同时还有一些方法用于在字节被解释为文本时提供有用的功能。在本节中，我们将讨论每种集合类型都具备的操作，比如创建、更新和读取数据。我们还将探讨`String`与其他集合的不同之处，即如何对`String`进行索引操作。由于人和计算机在解读`String`数据方面的差异，使得情况变得更加复杂。

<a id="什么是字符串"></a>

### 定义字符串

首先，我们来定义“字符串”这一术语的含义。在Rust的核心语言中，只有一个字符串类型，那就是通常被称为`str`的字符串切片，而它通常以另一种形式存在，即`&str`。在第四章中，我们讨论过字符串切片，它们是对存储在其他地方的一些UTF-8编码字符串数据的引用。例如，字符串字面量就存储在程序的二进制文件中，因此也可以被视为字符串切片。

`String`类型是由Rust的标准库提供，而不是硬编码到核心语言中的。它是一种可增长的、可变的、受拥有的、采用UTF-8编码的字符串类型。当Rust开发者在Rust中提到“字符串”时，他们可能指的是`String`类型或字符串切片`&str`类型，而不仅仅是其中的一种类型。虽然这一部分主要讨论的是`String`类型，但这两种类型都是相关的。在Rust的标准库中大量使用，而`String`和字符串切片都是UTF-8编码的。

```rust,ignore
fn add(self, s: &str) -> String {
```

在标准库中，你会看到 ``add`` 是通过泛型及相关类型定义的。在这里，我们替换了具体类型，当我们使用 ``String`` 的值调用此方法时，就会发生这种情况。我们将在第10章中讨论泛型。这个签名为我们理解 ``+`` 运算符的复杂部分提供了必要的线索。

首先，`s2`包含了一个`&`，这意味着我们正在将第二个字符串的引用添加到第一个字符串中。这是因为`add`函数中包含了`s`参数：我们只能将一个字符串切片添加到`String`中；而无法将两个`String`的值相加。不过，请注意——`&s2`的类型是`&String`，而不是`&str`。如`add`的第二个参数所指定的那样。那么，为什么Listing 8-18能够编译呢？

我们能够在使用`add`时调用`&s2`的原因在于，编译器可以将`&String`参数强制转换为`&str`。当我们调用`add`方法时，Rust会使用去引用转换机制，将`&s2`转换为`&s2[..]`。我们将在第十五章中更深入地讨论去引用转换机制。因为`add`并不接受这种转换。在进行了此操作之后，``s``参数的所有权仍然属于``s2``，而``String``仍然是一个有效的参数。

其次，从签名中可以看出，`add`负责处理`self`，因为`self`并没有`&`。这意味着清单8-18中的`s1`将会被移动到`add`中，之后它将不再有效。因此，尽管`let s3 = s1 + &s2;`看起来像是会复制两个字符串并创建一个新的字符串，但实际上这个语句会负责处理`s1`，并追加一个副本的内容。它属于`s2`，然后返回结果的拥有权。换句话说，看起来它似乎在创建了很多副本，但实际上并非如此；其实现方式比复制要高效得多。

在这种情况下，`len`将会变成`4`，这意味着存储该字符串的向量长度为4字节。在UTF-8编码下，每个字符需要1个字节来表示。不过，下面这一行可能会让你感到惊讶（请注意，这个字符串以大写西里尔字母_Ze_开头，而不是数字3）：

```rust
{{#rustdoc_include ../listings/ch08-common-collections/listing-08-14/src/main.rs:russian}}
```

如果你被问到字符串的长度，你可能会回答12。但实际上，Rust的答案是24：这是因为用UTF-8编码“Здравствуйте”需要24个字节，因为字符串中的每个Unicode标量值都需要2个字节的存储空间。因此，字符串的字节索引并不总能与有效的Unicode标量值相对应。为了说明这一点，请看这段无效的Rust代码：

```rust,ignore,does_not_compile
let hello = "Здравствуйте";
let answer = &hello[0];
```

您已经知道，`answer`不会是`З`，即第一个字母。当用UTF-8编码时，`З`的第一个字节是`208`，第二个字节是`151`。因此，看起来`answer`实际上应该是`208`。但是，`208`本身并不是一个有效的字符。如果用户输入的是这个字符串的第一个字母，那么返回`208`很可能不是他们想要的结果。然而，这是Rust在字节索引0处唯一能找到的数据。通常，用户并不希望得到字节值的返回，即使字符串中只包含拉丁字母。如果`&"hi"[0]`是能够返回字节值的有效代码，那么它应该返回`104`，而不是`h`。

因此，为了避免返回意外的值并导致可能不会被立即发现的错误，Rust根本不会编译这段代码，从而在开发过程的早期就避免了误解。

#### 字节、标量值以及图素簇

关于UTF-8的另一个要点是，从Rust的角度来看，实际上有三种相关的方式来查看字符串：作为字节、标量值，以及字符集（最接近我们所说的“字母”）。

如果我们看一下用天城文书写的印地语单词“नमस्ते”，它会被存储为一个由`u8`个值组成的向量，其结构如下：

```text
[224, 164, 168, 224, 164, 174, 224, 164, 184, 224, 165, 141, 224, 164, 164,
224, 165, 135]
```

这相当于18个字节，这也是计算机最终存储这些数据的方式。如果我们把它们视为Unicode标量值，也就是Rust中的`char`类型所表示的，那么这些字节看起来就是这样的：

```text
['न', 'म', 'स', '्', 'त', 'े']
```

这里有六个`char`值，但第四个和第六个并不是字母：它们实际上是用于标注的字符，单独来看是没有意义的。如果我们把它们视为一组字符，那么就能得到人们所说的构成印地语单词的四个字母。

```text
["न", "म", "स्", "ते"]
```

Rust提供了多种方式来解析计算机存储的原始字符串数据，这样每个程序都可以选择自己需要的解析方式，无论这些数据使用的是哪种人类语言。

最后，Rust不允许我们对`String`进行索引以获取单个字符的原因在于，索引操作应始终在常数时间内完成（O(1)）。但是，对于`String`来说，无法保证这种性能，因为Rust需要从头到尾遍历其内容，以确定有多少个有效的字符。

### 字符串切片

对字符串进行索引通常不是一个好主意，因为不清楚这种索引操作的返回类型应该是什么：是一个字节值、一个字符、一个字符群，还是一个字符串切片。因此，如果你真的需要使用索引来创建字符串切片，Rust会要求你提供更具体的说明。

与其使用单个数字来索引`[]`，不如使用`[]`并结合一个范围，从而创建一个包含特定字节的字符串切片。

```rust
let hello = "Здравствуйте";

let s = &hello[0..4];
```

在这里，`s`将是一个包含字符串前4个字节的`&str`。之前我们提到过，每个字符都是2个字节，这意味着`s`实际上就是`Зд`。

如果我们试图使用类似`&hello[0..1]`的工具来仅分割字符的某些字节，Rust会在运行时引发panic，就像在向量中访问无效索引时一样。

```console
{{#include ../listings/ch08-common-collections/output-only-01-not-char-boundary/output.txt}}
```

在创建带有范围的字符串切片时，应该非常小心，因为这样做可能会导致程序崩溃。

<a id="字符串遍历方法"></a>

### 遍历字符串

处理字符串的最佳方式是明确说明你是想要字符还是字节。对于单个Unicode标量值，可以使用`chars`方法。对“Зд”调用`chars`会将其分离出来，并返回一个类型为`char`的两个值，你可以遍历这个结果来访问每个元素。

```rust
for c in "Зд".chars() {
    println!("{c}");
}
```

这段代码将输出以下内容：

```text
З
д
```

或者，`bytes`方法会返回每个原始字节，这可能对你的领域来说比较合适：

```rust
for b in "Зд".bytes() {
    println!("{b}");
}
```

这段代码将打印出构成这个字符串的4个字节：

```text
208
151
208
180
```

但请务必记住，有效的Unicode标量值可能由超过1个字节组成。

从字符串中提取字符集群的过程与处理天城文一样复杂，因此标准库并不提供这一功能。如果你需要这个功能，可以在[crates.io](https://crates.io/)上找到相关包。

<a id="字符串并不那么简单"></a>

### 处理字符串中的复杂性

总结来说，字符串处理非常复杂。不同的编程语言在处理这种复杂性时会有不同的选择。Rust选择将所有Rust程序中的`String`数据都作为默认处理方式，这意味着程序员在处理UTF-8数据时需要更加注意。这种权衡使得字符串处理的复杂性比其他编程语言更为明显，但它也避免了其他语言可能带来的问题。在开发周期后期，需要处理与非ASCII字符相关的错误。

好消息是，标准库提供了许多功能，这些功能基于`String`和`&str`类型，可以帮助正确处理这些复杂的情况。请务必查阅相关文档，了解诸如用于字符串搜索的`contains`以及用于将字符串的部分内容替换为其他字符串的`replace`等有用的方法。
