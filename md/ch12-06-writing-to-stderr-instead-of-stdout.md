## 将错误重定向到标准错误

目前，我们所有的输出都是通过`println!`宏写入终端的。在大多数终端中，有两种类型的输出：用于一般信息的_标准输出__（`stdout`），以及用于错误信息的_标准错误__（`stderr`）。这种区分使用户可以选择将程序的成功输出保存到文件中，同时仍然可以将错误信息打印到屏幕上。

`println!`宏只能将输出到标准输出，因此我们必须使用其他方法将信息输出到标准错误。

### 检查错误所在的位置

首先，让我们观察一下`minigrep`所打印的内容是如何被写入标准输出的，以及我们是否希望将任何错误信息写入标准错误输出。为了实现这一点，我们将标准输出流重定向到一个文件中，同时故意引发一个错误。不过，我们不会将标准错误流重定向，因此所有发送到标准错误的内容仍然会显示在屏幕上。

命令行程序应该将错误信息发送到标准错误流中，这样即使我们将标准输出流重定向到文件中，我们仍然可以在屏幕上看到错误信息。不过，我们的程序目前表现不佳：看来它实际上是把错误信息输出保存到文件中了！

为了演示这种行为，我们将运行程序，同时使用``>``变量以及我们希望将标准输出流重定向到的文件路径`output.txt`。我们不会传递任何参数，这应该会导致一个错误：

```console
$ cargo run > output.txt
```

`>`语法告诉shell将标准输出内容写入output.txt_而不是屏幕。我们没有看到预期显示在屏幕上的错误信息，这意味着错误信息一定被记录到了文件中。这就是output.txt_中的内容：

```text
Problem parsing arguments: not enough arguments
```

是的，我们的错误信息被打印到标准输出上。对于这类错误信息来说，将其打印到标准错误输出会更实用，这样只有成功运行的数据才会被保存到文件中。我们会进行更改。
