## 不安全的Rust

到目前为止，我们所讨论的所有代码都遵循了Rust的内存安全保证，这些保证在编译时就会得到执行。不过，Rust内部还隐藏着另一种语言，它并不强制执行这些内存安全保证。这种语言被称为“不安全Rust”，它的运作方式与普通Rust相同，但赋予了我们使用额外的功能。

不安全的Rust之所以存在，是因为静态分析本质上是保守的。当编译器试图判断代码是否遵守了相关规范时，与其接受一些无效的代码，不如拒绝一些有效的代码。虽然这些代码可能看起来没有问题，但如果Rust编译器没有足够的信息来做出判断，它就会拒绝这些代码。在这种情况下，你可以使用不安全的代码来告诉编译器：“相信我，我知道自己在做什么。”不过，需要注意的是，使用不安全的Rust代码需要您自行承担风险：如果您错误地使用了不安全的代码，可能会因为内存安全问题而出现问题，例如空指针解引用。

Rust拥有“不安全”版本的另一个原因是，底层的计算机硬件本身就不安全。如果Rust不允许进行不安全的操作，那么就无法完成某些任务。Rust需要允许开发者进行底层系统编程，比如直接与操作系统交互，甚至编写自己的操作系统。进行底层系统编程正是该语言的目标之一。让我们来探讨一下如何使用Rust的“不安全”版本来实现这些功能。

### 使用不安全的超级能力

要切换到不安全的Rust环境，可以使用`unsafe`这个关键字，然后开始一个新的代码块，其中包含了不安全的代码。在不安全的Rust环境中，你可以执行一些在安全Rust中无法进行的操作，我们将这些操作称为“不安全的超能力”。这些超能力包括：

1. 解引用原始指针。  
1. 调用不安全的函数或方法。  
1. 访问或修改可变的静态变量。  
1. 实现不安全的特性。  
1. 访问`union`s的字段。

需要明白的是，`unsafe`并不会关闭借用检查器，也不会禁用Rust的其他安全检查机制。如果你在unsafe代码中使用了引用，这些引用仍然会被进行检查。而`unsafe`这个关键字只是让你能够访问这五个功能，但这些功能不会由编译器进行内存安全性的检查。尽管如此，你在unsafe代码块内仍然可以获得一定程度的安全性保障。

此外，`unsafe`并不意味着该代码块中的内容一定具有危险性，或者一定会存在内存安全方面的问题。其目的是让程序员能够确保位于`unsafe`代码块中的代码能够以合法的方式访问内存。

人们都会犯错，错误是难免的，但是通过要求这五种不安全操作必须位于带有 ``unsafe`` 注释的块内，你就可以确保与内存安全相关的任何错误都发生在 ``unsafe`` 块内。尽量保持 ``unsafe`` 块的小规模；这样在排查内存漏洞时你会感到庆幸。

为了尽可能隔离不安全的代码，最好将这些代码封装在一个安全的抽象层中，并提供安全的API。我们将在后面章节中讨论如何对不安全的函数和方法进行处理。标准库中的某些部分实际上是基于经过审核的不安全代码实现的安全抽象层。将不安全的代码封装在安全的抽象层中，可以防止``unsafe``被泄露到您或您的用户可能想要使用的所有地方。使用`unsafe`代码实现了该功能，因为使用safeabstraction是安全的。

让我们依次来看看这五个不安全的超级权限。我们还会探讨一些能够为不安全代码提供安全接口的抽象概念。

### 解引用原始指针

在第四章的“悬空引用”部分中，我们提到编译器会确保引用始终有效。不安全的Rust语言中有两种新类型，称为“原始指针”，它们与引用类似。与引用一样，原始指针可以是不可变的，也可以是可变的，并且分别被表示为`*const T`和`*mut T`。星号并不是解引用运算符；它是类型名称的一部分。在原始指针的上下文中……指针，_不可变_意味着在解引用之后，该指针不能被直接赋值给另一个变量。

与引用和智能指针不同，原始指针：

- 允许使用不可变指针和可变指针或多个可变指针来忽略借用规则，从而指向同一位置
- 不保证会指向有效的内存地址
- 允许为空值
- 不实现任何自动清理机制

通过选择不让Rust执行这些保证，你可以放弃有保障的安全性，以换取更高的性能，或者能够与其他语言或硬件进行交互，而Rust的保证在这些情况下并不适用。

#### 使用其他语言调用Rust函数

我们还可以使用`extern`来创建一个接口，使得其他语言能够调用Rust函数。无需创建完整的`extern`块，我们可以在`fn`关键字之前添加`extern`关键字，并指定要使用的ABI。同时，我们还需要添加`#[unsafe(no_mangle)]`注释，以告知Rust编译器不要修改该函数的名称。_修改_当编译器将我们给函数的名称改为另一个名称时，这个名称会包含更多信息，这些信息会被编译过程的其他部分使用，但可读性较差。每种编程语言的编译器对名称的处理方式都略有不同。因此，为了让Rust函数能够被其他语言命名，我们必须禁用Rust编译器的名称处理功能。这样做是不安全的，因为不同库之间可能会出现名称冲突。进行混淆处理，因此我们的责任是确保所选择的名称在导出过程中不会受到混淆影响，保持安全性。

在下面的示例中，我们使得在C代码中对`call_from_c`函数的访问成为可能，这一过程包括将其编译成共享库，并从C代码中进行链接。

```
#[unsafe(no_mangle)]
pub extern "C" fn call_from_c() {
    println!("Just called a Rust function from C!");
}
```

这种使用`extern`的方式要求`unsafe`仅存在于属性中，而不出现在`extern`块中。

常数与不可变的静态变量之间的一个微妙区别在于，静态变量中的值在内存中有固定的地址。使用这些值时，总是可以访问相同的数据。而常数则可以在使用时随时复制其数据。另一个区别是，静态变量可以是可变的。访问和修改可变的静态变量是不安全的。清单20-11展示了如何声明、访问和修改一个名为`COUNTER`的可变静态变量。

与常规变量一样，我们使用``mut``关键字来指定变量的可变性。任何从``COUNTER``读取或写入数据的代码都必须位于``unsafe``块内。正如我们所预期的那样，清单20-11中的代码能够编译并打印出``COUNTER: 3``，因为它是在单线程环境下运行的。如果多个线程访问``COUNTER``，可能会导致数据竞争，因此这属于未定义的行为。因此，我们需要采取适当的措施来确保线程安全。将整个函数标记为 ``unsafe``，并详细说明安全限制，以便调用该函数的任何人都能了解允许和禁止做的事情。

每当我们编写不安全的函数时，通常会使用如下形式的注释来说明：以`SAFETY`开头，并解释调用者需要做什么才能安全地调用该函数。同样地，每当我们进行不安全的操作时，也会使用类似的形式的注释来说明如何遵守安全规则。

此外，编译器默认会拒绝任何通过编译器的检查来创建对可变静态变量的引用的尝试。您必须明确选择放弃这些保护措施，方法是添加`#[allow(static_mut_refs)]`注解，或者通过一些原始借用运算符创建的原始指针来访问可变静态变量。这包括那些以不可见方式创建引用的情况，比如在代码中使用这样的引用时。在这段代码清单中使用了`println!`。要求通过原始指针来创建静态可变变量，这有助于使使用这些变量的安全要求更加明确。

由于数据是可变的且可以全局访问，因此很难确保不会出现数据竞争的情况。这就是为什么Rust认为可变的静态变量是不安全的。在可能的情况下，最好使用我们在第16章中讨论的并发技术以及线程安全的智能指针，这样编译器就能检查来自不同线程的数据访问是否安全进行。

通过使用`unsafe impl`，我们承诺会遵守那些编译器无法验证的不变量。

例如，回想一下我们在第16章的“使用`Send`和`Sync`实现可扩展并发”部分中提到的`Send`和`Sync`标记特性。如果我们实现的类型完全由其他实现了`Send`和`Sync`的类型组成，那么编译器会自动应用这些特性。但是，如果我们实现的一种类型中包含一种未实现这些特性的类型……`Send`或`Sync`这样的原始指针类型，如果我们希望将其标记为`Send`或`Sync`类型，那么我们必须使用`unsafe`。Rust无法验证我们的类型是否能够满足跨线程安全传输或从多个线程中访问的要求；因此，我们需要手动进行这些检查，并通过`unsafe`来标明这一点。

### 访问联合字段

最后一个适用于`unsafe`的操法是访问联合体的字段。  
“联合体”类似于`struct`，但在特定情况下，一次只能使用一个已声明的字段。联合体主要用于与C代码中的联合体进行交互。访问联合体的字段是不安全的，因为Rust无法保证当前存储在联合体中的数据的类型。您可以在[Rust参考手册][unions]中了解更多关于联合体的信息。

### 使用Miri检查不安全代码

在编写不安全的代码时，你可能需要检查自己所写的代码是否真正安全且正确。实现这一目标的最佳方法之一是使用Miri，这是Rust官方提供的一个用于检测未定义行为的工具。虽然borrow checker是一个在编译时生效的静态工具，但Miri则是一个在运行时生效的动态工具。它通过运行你的程序或其测试套件来检测你的代码，并识别出你违反了Rust规范所规定的规则。

使用Miri需要安装最新版本的Rust（我们在[附录G：Rust的制造过程与“Nightly Rust”](http://nightly)中有更多介绍）。您可以通过运行`rustup +nightly component add miri`来安装最新版本的Rust以及Miri工具。这一操作并不会改变项目所使用的Rust版本；它只是将Miri工具添加到您的系统中，以便您可以在需要时使用它。您可以通过输入`cargo +nightly miri run`或`cargo +nightly miri test`来运行Miri工具。

### 正确使用不安全代码

使用 ``unsafe`` 来使用刚才提到的五种超能力之一，这本身并没有错，甚至也不被提倡。但是，要确保 ``unsafe`` 代码的正确性则更为复杂，因为编译器无法完全保证内存安全。当你有理由使用 ``unsafe`` 时，你可以这样做。而使用明确的 ``unsafe`` 注释，则有助于更容易地追踪问题发生的根源。编写不安全的代码时，可以使用Miri来帮助你更放心地编写代码，确保你所写的代码符合Rust的规范。

如果您想更深入地了解如何有效地使用不安全的Rust，请阅读Rust的官方指南《Rust’s official guide for `unsafe`》，以及《The Rustonomicon》[nomicon]。
