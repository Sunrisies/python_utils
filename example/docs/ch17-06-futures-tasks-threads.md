## 综合概述：未来、任务与线程

正如我们在 [Chapter 16][ch16]<!-- ignore --> 中看到的，线程提供了一种实现并发的方式。在本章中，我们还介绍了另一种实现并发的方法：使用异步函数和流。如果你想知道何时应该选择其中一种方法，答案是：这取决于具体情况！在许多情况下，选择不是线程_或_异步，而是线程_和_异步。

许多操作系统已经以基于线程的并发模型为基础运行了数十年，因此许多编程语言也支持这些模型。然而，这些模型并非没有缺点。在许多操作系统中，每个线程会占用相当多的内存资源。而且，线程只有在操作系统和硬件支持的情况下才可用。与主流台式机和移动计算机不同，一些嵌入式系统根本就没有操作系统，因此也就没有线程。

异步模型提供了一组不同的——且最终是互补的——权衡方案。在异步模型中，并发操作不需要自己的线程。相反，它们可以在任务上运行，就像我们在 streams 部分使用 `trpl::spawn_task` 来从同步函数开始执行工作那样。任务类似于线程，但与其由操作系统管理，不如说是由库级代码——即运行时——来管理的。

线程和任务相关的 API 之所以如此相似，是有原因的。线程充当同步操作集合的边界；并发可以在线程之间实现。而任务则充当异步操作集合的边界；并发既可以在任务之间实现，也可以在任务内部实现，因为任务可以在其执行过程中切换不同的未来操作。最后，未来操作是 Rust 中最细粒度的并发单位，每个未来操作可能代表着其他未来操作的树状结构。运行时——具体来说，就是它的执行器——负责管理任务，而任务则负责管理未来操作。从这个角度来看，任务类似于由运行时管理的轻量级线程，只不过它们还具备一些额外的功能，这些功能是由运行时而不是操作系统来管理的。

这并不意味着异步任务总是优于线程（或者相反）。在某些方面，使用线程进行并发处理比使用 `async` 进行并发处理更为简单。这既可以是优势，也可能是劣势。线程的运作方式较为“点火后即可放弃”；它们没有类似于未来的概念，因此线程会一直运行到结束，除非被操作系统本身中断。

事实证明，线程和任务通常可以很好地协同工作，因为任务可以在某些运行时环境中在多个线程之间移动。实际上，在我们所使用的运行时环境中——包括 `spawn_blocking` 和 `spawn_task` 函数——默认情况下就是多线程的！许多运行时环境采用一种称为“工作窃取”的方法，根据线程当前的使用情况来透明地移动任务，从而提高系统的整体性能。这种方法实际上需要线程以及任务，因此也需要使用未来函数。

在考虑使用哪种方法时，可以参考以下经验法则：

- 如果任务具有**高度并行性**（即依赖CPU处理），比如处理大量数据，且每个部分都可以单独处理，那么使用线程是一个更好的选择。
- 如果任务具有**高度并发性**（即依赖I/O操作），比如处理来自多个不同来源的消息，这些消息可能以不同的时间间隔或速率到达，那么使用异步编程是一个更好的选择。

如果你需要并行性和并发性，你不必在线程和异步之间做出选择。你可以自由地将它们结合起来使用，让每个技术特性发挥它最擅长的功能。例如，清单17-25展示了现实世界Rust代码中这种混合使用的常见例子。

<Listing number="17-25" caption="Sending messages with blocking code in a thread and awaiting the messages in an async block" file-name="src/main.rs">

```rust
use std::{thread, time::Duration};

fn main() {
    let (tx, mut rx) = trpl::channel();

    thread::spawn(move || {
        for i in 1..11 {
            tx.send(i).unwrap();
            thread::sleep(Duration::from_secs(1));
        }
    });

    trpl::block_on(async {
        while let Some(message) = rx.recv().await {
            println!("{message}");
        }
    });
}

```

</Listing>

我们首先创建一个异步通道，然后创建一个线程，该线程使用 `move` 关键字来负责通道的发送端。在該线程中，我们依次发送数字 1 到 10，每次发送之间休息一秒。最后，我们运行一个由 async 块创建的未来对象，该对象通过 `trpl::block_on` 传递给线程。在该未来中，我们等待这些消息的到来，就像我们在本章中看到的其他消息传递示例一样。

回到我们开始讨论的场景，假设我们使用一个专用线程来执行视频编码任务（因为视频编码需要大量的计算资源），同时通过异步通道向用户界面通知这些操作已经完成。在现实世界的用例中，这种组合方式有很多例子。
