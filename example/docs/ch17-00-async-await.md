# 异步编程基础：异步、等待、未来值和流

我们要求计算机执行的许多操作可能需要一段时间才能完成。如果我们能在等待这些耗时较长的进程完成的同时，能够做其他事情的话，那该多好。现代计算机提供了两种技术来同时处理多个操作：并行性和并发性。然而，我们的程序逻辑大多是线性编写的。我们希望能够指定程序应该执行的操作，并指定函数可以暂停的位置，让程序的其他部分能够替代地运行，而无需事先明确指定每部分代码应该以何种顺序和方式运行。异步编程是一种抽象机制，它让我们能够用潜在的暂停点和最终结果来表达我们的代码，从而为我们处理协调的细节问题。

本章在第十六章关于线程并行与并发的处理基础上，介绍了一种编写代码的替代方法：Rust中的未来、流，以及 `async` 和 `await` 语法，这些语法可以帮助我们表达操作的异步执行方式。此外，还介绍了一些第三方库，它们实现了异步运行时环境，能够管理和协调异步操作的执行。

让我们来看一个例子。假设您正在导出自己拍摄的家庭庆祝活动的视频，这个过程可能需要几分钟到几小时不等。视频导出会尽可能多地消耗CPU和GPU资源。如果您只有单个CPU核心，而且操作系统在导出完成之前不会暂停该进程——也就是说，如果导出是同步执行的——那么在任务进行期间，您就无法在电脑上执行其他任何操作。这确实会是一种非常令人沮丧的体验。幸运的是，您的电脑操作系统能够并且确实会频繁地中断导出过程，从而让您能够同时完成其他任务。

现在假设你正在下载其他人分享的视频，这个过程同样需要一些时间，但所消耗的CPU时间并不多。在这种情况下，CPU必须等待数据从网络传输过来。虽然可以在数据开始传输后立即开始读取数据，但所有数据都传输完毕可能需要一段时间。即便数据已经全部传输完毕，如果视频文件体积较大，加载整个视频至少也需要一两秒钟。这听起来可能不算什么，但对于现代处理器来说，这可是相当长的时间——现代处理器每秒可以执行数十亿次操作。同样，你的操作系统会无形中中断程序运行，让CPU在等待网络调用完成的同时执行其他任务。

视频导出是一种_受CPU限制_或_受计算能力限制_的操作。这种操作受到计算机在CPU或GPU上处理数据的速度的限制，以及计算机能够将这些速度中的多少部分用于该操作。而视频下载则是一种_受I/O限制_的操作，因为它受到计算机_输入和输出_速度的限制；其速度只能与数据通过网络传输的速度相当。

在这两个例子中，操作系统的不可见中断提供了一种并发机制。不过，这种并发只发生在整个程序的层面上：操作系统会中断一个程序，以便让其他程序能够继续运行。在许多情况下，由于我们对程序的理解比操作系统更为细致，因此我们能够发现操作系统无法察觉的并发机会。

例如，如果我们正在开发一个用于管理文件下载的工具，那么我们应该能够编写程序，使得开始一个下载操作不会锁定用户界面，同时用户也可以同时启动多个下载任务。不过，许多操作系统用于与网络交互的API都是采用“阻塞”方式的；也就是说，它们会阻塞程序的执行，直到正在处理的数据完全准备好为止。

> 注意：大多数函数调用的运作方式都是如此，只要您仔细思考一下就会明白。不过，术语“阻塞”通常只用于那些与文件、网络或计算机上的其他资源交互的函数调用，因为这些情况下，单个程序会受益于非阻塞的操作。

我们可以通过创建一个专门的线程来下载每个文件，从而避免阻塞主线程。然而，这些线程所使用的系统资源所带来的开销最终会成为一个问题。理想的情况是，调用操作本身就不应该阻塞，而是我们可以定义一系列任务，让运行时能够选择最佳的排序和执行方式。

这正是Rust的异步抽象功能所提供的能力。在本章中，你将学习有关异步的所有内容，我们将涵盖以下主题：

- 如何使用 Rust 的 `async` 和 `await` 语法，以及如何通过运行时来执行异步函数。  
- 如何使用异步模型来解决我们在第 16 章中讨论过的一些挑战。  
- 多线程和异步如何提供互补的解决方案，这些解决方案在许多情况下可以被结合起来使用。

不过，在了解异步在实际中的工作原理之前，我们需要先简要讨论一下并行性和并发性之间的区别。

## 并行性与并发性

到目前为止，我们一直认为并行性和并发性是可以互换的。但现在我们需要更精确地区分它们，因为随着工作的进行，这些差异会逐渐显现出来。

考虑一个团队在软件项目中分配工作的不同方式。你可以让一名成员负责多个任务，或者让每名成员负责一个任务，也可以结合这两种方式来进行分配。

当一个人同时处理多个不同的任务，而其中任何一个任务尚未完成时，这就是_并发_。实现并发的一种方式类似于在电脑上同时打开两个不同的项目：当你对其中一个项目感到厌倦或陷入困境时，就可以切换到另一个项目。由于你只是一个人，因此无法同时在两个任务上取得进展，但你可以进行多任务处理，通过在不同任务之间切换来逐步完成其中一个任务（参见图17-1）。

在这两种工作流程中，你可能需要协调不同的任务。也许你认为分配给某个人的任务与其他人的工作完全独立，但实际上需要团队中的另一个人先完成他们的任务。有些工作可以并行进行，但有些工作实际上是顺序进行的——它们只能一个接一个地完成，就像图17-3所示那样。

同样，你可能会发现自己的某个任务依赖于另一个任务。现在，你的并发工作也变成了串行操作。

并行性和并发性也可以相互交织。如果你发现某个同事因为你的任务未完成而陷入困境，你很可能会全力以赴地帮助他，以“解决”他的难题。这样一来，你和你的同事就无法同时进行工作了，你自己处理的任务也变得无法并行执行了。

软件和硬件也遵循同样的基本规律。在只有单个CPU核心的机器上，CPU一次只能执行一个操作，但它仍然可以同时进行多个任务。通过使用线程、进程和异步等工具，计算机可以暂停一个任务，转而执行其他任务，最终再回到最初的任务上。在拥有多个CPU核心的机器上，任务也可以并行执行。一个核心可以执行一个任务，而另一个核心则可以执行完全不相关的任务，这些操作实际上可以同时发生。

在Rust中运行异步代码通常是通过并发方式实现的。根据硬件、操作系统以及所使用的异步运行时（关于异步运行时的内容稍后介绍），这种并发处理可能会利用并行处理机制来加速执行速度。
