## 无法恢复的 `panic!` 错误

有时候，你的代码中会出现一些糟糕的情况，而对此你无能为力。在这种情况下，Rust提供了`panic!`宏。实际上，有两种方式可以引发panic：一种是执行某个会导致程序崩溃的操作（比如访问数组的末尾元素），另一种则是显式调用`panic!`宏。在这两种情况下，程序都会引发panic。默认情况下，这些panic会打印出错误信息，然后解除锁，清理栈，并退出程序。你也可以通过环境变量让Rust在发生panic时显示调用栈，这样就能更容易地追踪panic的来源。

> ### 在出现 panic 时，程序会进行栈回溯或立即终止
>
> 默认情况下，当发生 panic 时，程序会开始进行栈回溯操作，即 Rust 会沿着栈的线索返回，并清理每个函数所占用的数据。然而，这种回溯和清理过程非常耗时。因此，Rust 允许你选择立即终止程序，从而无需进行清理操作。
>
> 此时，程序所使用的内存将由操作系统负责清理。如果你希望生成的二进制文件尽可能小，可以在 _Cargo.toml_ 文件中，在相应的 `[profile]` 部分添加 `panic = 'abort'`，以便在出现 panic 时直接终止程序。例如，如果你想在发布模式下在出现 panic 时立即终止程序，可以添加如下内容：
>
> ```toml
> [profile.release]
> panic = 'abort'
> ```

让我们在一个简单的程序中尝试调用 `panic!`：

<Listing file-name="src/main.rs">

```rust,should_panic,panics
fn main() {
    panic!("crash and burn");
}

```

</Listing>

当你运行这个程序时，你会看到类似这样的结果：

```console
$ cargo run
   Compiling panic v0.1.0 (file:///projects/panic)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.25s
     Running `target/debug/panic`

thread 'main' panicked at src/main.rs:2:5:
crash and burn
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

```

调用 `panic!` 会导致最后两行出现错误信息。第一行显示了我们的恐慌信息，以及恐慌发生的位置：_src/main.rs:2:5_ 表示这是 _src/main.rs_ 文件的第二行，第五个字符的位置。

在这种情况下，所指示的那一行是我们代码的一部分。如果我们进入那一行代码，就会看到 `panic!` 宏的调用。在其他情况下， `panic!` 的调用可能出现在我们代码调用的其他代码中，而错误信息中报告的文件名和行号则属于其他人编写的代码中，在那里使用了 `panic!` 宏，而不是最终导致 `panic!` 调用的那一行代码。

<a id="using-a-panic-backtrace"></a>

我们可以利用 `panic!` 调用的函数的调用链来找出导致问题的代码部分。为了了解如何使用 `panic!` 调用链，让我们再看一个例子，看看当 `panic!` 调用来自库中的代码时的情况，这种情况是由于代码中的错误导致的，而不是因为代码直接调用了宏。列表 9-1 中有一段代码，它试图访问向量中超出有效索引范围的索引。

<Listing number="9-1" file-name="src/main.rs" caption="Attempting to access an element beyond the end of a vector, which will cause a call to `panic!`">

```rust,should_panic,panics
fn main() {
    let v = vec![1, 2, 3];

    v[99];
}

```

</Listing>

在这里，我们试图访问向量中的第100个元素（因为索引是从0开始的，所以实际上是第99个元素），但向量实际上只有三个元素。在这种情况下，Rust会抛出panic异常。使用 `[]` 本应返回一个元素，但如果你传递了一个无效的索引，Rust无法返回任何正确的元素。

在C语言中，试图读取超过数据结构末尾的数据会导致未定义的行为。你可能会读取到内存中与该数据结构中的元素相对应的数据，尽管该内存并不属于该数据结构。这种现象被称为“缓冲区越读”，如果攻击者能够操纵索引，从而读取到不应该被允许的数据，那么就可能引发安全漏洞。

为了保护你的程序免受这种漏洞的影响，如果你尝试读取一个不存在的索引处的元素，Rust会停止执行并拒绝继续运行。让我们试试看：

```console
$ cargo run
   Compiling panic v0.1.0 (file:///projects/panic)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.27s
     Running `target/debug/panic`

thread 'main' panicked at src/main.rs:4:6:
index out of bounds: the len is 3 but the index is 99
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

```

这个错误出现在我们的 _main.rs_ 文件的第4行，那里我们试图访问 `v` 中向量的第99个索引。

The `note:`这一行告诉我们，我们可以设置 `RUST_BACKTRACE` 环境变量，以获取导致错误的确切原因。一个 _backtrace_ 是一个列表，其中包含了所有被调用以到达这一点的函数。在Rust中，回溯的实现与其他语言相同：阅读回溯的关键是从顶部开始，一直读到你编写过的代码。那些位于问题起始点的行，就是你的代码调用的代码；而位于其下方的行，则是调用你代码的代码。这些前后对比的行可能包含Rust的核心代码、标准库代码，或者你正在使用的其他库。让我们尝试通过设置 `RUST_BACKTRACE` 环境变量为除 `0` 之外的任何值来获取回溯信息。列表9-2显示了与你看到的类似输出。

在 Listing 9-2 的输出中，回溯路径的第 6 行指向了我们项目中导致问题的那一行代码，即 _src/main.rs_ 的第 4 行。如果我们不希望程序出现 panic 情况，那么我们应该从第一行所指向的位置开始调查，该位置所指向的文件是我们自己编写的。在 Listing 9-1 中，我们故意编写了会导致 panic 的代码，解决 panic 的方法就是不要请求超出向量索引范围的元素。当你的代码将来出现 panic 时，你需要弄清楚代码正在使用哪些值来引发 panic，以及代码应该采取什么措施来避免这种情况。
