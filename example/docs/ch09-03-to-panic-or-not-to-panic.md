## 是应该属于 `panic!` 还是不属于 `panic!`

那么，如何决定何时调用 `panic!`，何时返回 `Result` 呢？当代码出现故障时，是无法恢复的。你可以为任何错误情况调用 `panic!`，无论是否有办法恢复，但这样你就代表调用代码来决定某种情况是无法恢复的。当你选择返回 `Result` 的值时，你为调用代码提供了选择的机会。调用代码可以选择以适合其情况的方式尝试恢复，或者决定在这种情况下 `Err` 的值是无法恢复的，从而可以调用 `panic!`，将你的可恢复错误转变为不可恢复的错误。因此，在定义可能失败的函数时，返回 `Result` 是一个很好的默认选择。

在示例、原型代码和测试等场景中，编写会导致程序崩溃的代码更为合适，而不是返回 `Result`。接下来，我们将探讨为什么如此做，并讨论那些编译器无法判断出失败是不可能的，但作为人类可以感知的情况。本章最后将提供一些关于如何在库代码中决定是否使用崩溃处理的通用指导原则。

### 示例、原型代码与测试

在编写示例来阐释某个概念时，如果包含了鲁棒性的错误处理代码，那么示例的清晰度就会降低。在示例中，调用像 `unwrap` 这样的方法会被视为一个占位符，用来表示应用程序处理错误的方式。实际上，错误处理的方式可能会因代码中其他部分的操作而有所不同。

同样，当您在进行原型设计时，还不确定如何处理错误时，`unwrap`和`expect`方法非常实用。这些方法会在代码中留下明确的标记，以便您将来能够让程序更加健壮。

如果某个方法在测试中失败，你希望整个测试都失败，即使该方法并不是要测试的功能。因为`panic!`是用来标记测试失败的，所以调用`unwrap`或`expect`正是应该发生的情况。

### 当您拥有的信息比编译器所知道的更多时

在存在其他逻辑来确保 `Result` 会有一个 `Ok` 值的情况下，调用 `expect` 也是合适的。不过，这种逻辑并不是编译器能够理解的。你仍然需要处理 `Result` 这个值：无论你调用的是什么操作，通常都有失败的可能性，尽管在你的特定情况下这种失败在逻辑上是不可能的。如果你通过手动检查代码来确保永远不会出现 `Err` 这种变体，那么调用 `expect` 并在文档中说明你认为永远不会出现 `Err` 这种变体的原因是完全可以接受的。以下是一个例子：

```rust
    use std::net::IpAddr;

    let home: IpAddr = "127.0.0.1"
        .parse()
        .expect("Hardcoded IP address should be valid");

```

我们通过解析一个硬编码的字符串来创建一个 `IpAddr` 实例。我们可以看到 `127.0.0.1` 是一个有效的 IP 地址，因此在这里使用 `expect` 是可行的。然而，拥有一个硬编码且有效的字符串并不会改变 `parse` 方法的返回类型：我们仍然会得到一个 `Result` 值，而且编译器会要求我们像处理 `Err` 这种可能性一样来处理 `Result`，因为编译器没有足够的智能来识别这个字符串始终是一个有效的 IP 地址。如果 IP 地址字符串是由用户提供的，而不是硬编码到程序中，那么肯定会有失败的可能性，在这种情况下，我们当然希望以更稳健的方式来处理 `Result`。提及这个 IP 地址是硬编码的这一假设，会促使我们在将来需要从其他来源获取 IP 地址时，将 `expect` 更改为更好的错误处理代码。

### 错误处理指南

建议在代码可能陷入不良状态时触发 panic 机制。在这种情况下，“不良状态”指的是某些假设、保证、契约或不变性被打破的情况，比如当无效值、矛盾的值或缺失的值被传递给代码时——此外还可能出现以下一种或多种情况：

- 糟糕的状态是一种出乎意料的状况，而不是像用户以错误的格式输入数据这样的偶尔会发生的情况。
- 在此之后，你的代码需要依赖于不会进入这种糟糕的状态，而不是在每一步都检查是否存在问题。
- 你使用的类型并没有很好的方式来编码这些信息。我们将在第十八章的[“将状态和行为编码为类型”][encoding]<!-- ignore -->这一节中通过一个例子来说明我们的意思。

如果有人调用你的代码并传入一些没有意义的值，最好能返回一个错误，这样库的使用者就可以决定在这种情况下该怎么做。然而，在继续运行可能会带来安全隐患或损害的情况下，最好的选择可能是调用 `panic!`，并提醒使用你库的人注意他们代码中的错误，以便他们在开发过程中进行修复。同样，如果你调用的是外部代码，而这些代码不受你的控制，并且返回了无法修复的无效状态，那么调用 `panic!` 通常是一个合适的选择。

然而，当故障是预期之中的时候，返回 `Result` 比调用 `panic!` 更为合适。例如，当解析器接收到损坏的数据，或者HTTP请求返回的状态表明已经达到了速率限制时。在这些情况下，返回 `Result` 意味着故障是一种预期中的可能性，调用代码必须自行决定如何处理。

当您的代码执行某个操作，而使用无效的值调用该操作可能会让用户面临风险时，您应该先验证这些值是否有效，如果无效则触发panic。这主要是出于安全考虑：尝试处理无效的数据可能会使代码暴露于漏洞风险之中。这也是为什么标准库会在您尝试进行越界内存访问时调用 `panic!`：试图访问不属于当前数据结构的内存是一种常见的安全问题。函数通常都有契约：只有当输入满足特定要求时，函数的行为才是可靠的。当违反契约时触发panic是合理的，因为契约违反总是表明调用方存在错误，而这不是调用代码需要显式处理的错误类型。实际上，调用代码没有合理的办法来恢复这种状况；需要由调用方的技术人员来修复代码。对于函数的契约，尤其是当违反契约会导致panic时，应该在该函数的API文档中进行说明。

不过，如果在所有函数中都进行大量的错误检查，不仅会使得代码变得冗长且令人烦恼。幸运的是，你可以利用 Rust 的类型系统（以及编译器所进行的类型检查），来自动完成许多检查工作。如果你的函数有一个特定的类型作为参数，那么你可以按照自己的逻辑编写代码，而无需担心编译器已经确保了输入的值是有效的。例如，如果你使用的是类型而不是 `Option`，那么程序期望得到的是某种值，而不是空值。这样，你的代码就不需要处理 `Some` 和 `None` 这两种情况：它只需要处理一种存在有效值的情况即可。试图向函数传递空值的代码根本无法编译，因此函数在运行时无需检查这种情况。另一个例子是使用像 `u32` 这样的无符号整数类型，这种类型可以确保参数永远不会是负值。

### 用于验证的自定义类型

让我们进一步利用 Rust 的类型系统来确保我们获得的是有效的数值，然后来看看如何创建用于验证的自定义类型。回想一下第 2 章中的“猜数游戏”，我们的代码要求用户猜测 1 到 100 之间的数字。在检查用户的猜测是否与我们的秘密数字匹配之前，我们从未验证过用户的猜测是否在范围内；我们只是验证了猜测的数值是否为正数。在这种情况下，后果并不严重：我们输出“太高”或“太低”仍然是正确的。但是，如果能引导用户做出有效的猜测，并且在用户输入超出范围的数字时与输入字母时采取不同的处理方式，那将是一个很好的改进。

一种方法是将“guess”解析为 `i32`，而不仅仅是 `u32`，这样就能支持可能是负数的情况。然后，需要添加一个检查，确保该数字在指定范围内，具体做法如下：

```rust,ignore

        // --snip--

        let guess: i32 = match guess.trim().parse() {
            Ok(num) => num,
            Err(_) => continue,
        };

        if guess < 1 || guess > 100 {
            println!("The secret number will be between 1 and 100.");
            continue;
        }

        match guess.cmp(&secret_number) {
            // --snip--
    }

```

</Listing>

`if`表达式用于检查我们的数值是否超出了范围，并向用户提示存在的问题。之后会调用`continue`来开始循环的下一次迭代，并再次请求用户输入一个猜测值。在`if`表达式之后，我们可以继续比较`guess`与秘密数字之间的结果，同时知道`guess`的值应该在1到100之间。

然而，这并不是理想的解决方案：如果程序必须仅处理1到100之间的数值，并且有许多函数都符合这一要求，那么在每个函数中都加入这样的检查将会非常繁琐（而且可能会影响性能）。

相反，我们可以在专门的模块中创建一个新的类型，并将验证功能放在一个函数中，用于创建该类型的实例，而不是在各处重复这些验证。这样，函数可以在其签名中使用新类型，并且可以放心地使用接收到的数值。列表9-13展示了一种定义 `Guess` 类型的方法，该类型只有在 `new` 函数接收到的数值在1到100之间时，才会创建 `Guess` 类型的实例。

<Listing number="9-13" caption="A `Guess` type that will only continue with values between 1 and 100" file-name="src/guessing_game.rs">

```rust
pub struct Guess {
    value: i32,
}

impl Guess {
    pub fn new(value: i32) -> Guess {
        if value < 1 || value > 100 {
            panic!("Guess value must be between 1 and 100, got {value}.");
        }

        Guess { value }
    }

    pub fn value(&self) -> i32 {
        self.value
    }
}

```

</Listing>

请注意，*src/guessing_game.rs* 中的代码依赖于在 *src/lib.rs* 中添加了一个模块声明 `mod guessing_game;`，这一点我们在这里并未展示。在这个新模块的文件内，我们定义了一个名为 `Guess` 的结构体，该结构体有一个名为 `value` 的字段，该字段包含一个 `i32`。数字将被存储在这个字段中。

然后，我们在 `Guess` 上实现了一个名为 `new` `Guess` 的关联函数，该函数会创建 `Guess` 的实例。 `new` 函数被定义为接受一个名为 `value` 的参数，该参数的类型为 `i32`，并且返回一个 `Guess`。 `new` 函数的主体部分会测试 `value`，确保其值在 1 到 100 之间。如果 `value` 未能通过此测试，我们将调用 `panic!`，这会提醒编写调用代码的程序员他们有一个需要修复的错误，因为在此范围之外使用 `value` 创建 `Guess` 会违反 `Guess::new` 所依赖的契约。 `Guess::new` 可能触发恐慌的条件应在其公开 API 文档中讨论；我们将在第 14 章中介绍文档编写规范，说明 API 文档中可能存在 `panic!` 的情况。如果 `value` 通过了测试，我们将创建一个新的 `Guess`，并将其 `value` 字段设置为 `value` 参数，然后返回 `Guess`。

接下来，我们实现了一个名为 `value` 的方法。该方法继承了 `self` 的功能，没有任何其他参数，并且返回一个 `i32`。这种方法有时被称为 _getter_，因为它的作用是从其字段中获取一些数据并返回。这个公共方法是必要的，因为 `Guess` 结构体中的 `value` 字段是私有的。确保 `value` 字段是私有的非常重要，这样使用 `Guess` 结构体的代码就无法直接设置 `value`。 `guessing_game` 模块之外的代码必须使用 `Guess::new` 函数来创建 `Guess` 的实例，从而确保没有 `Guess` 能够拥有未经 `Guess::new` 函数中条件检查的 `value`。

一个具有参数或仅返回1到100之间数字的函数，可以在其签名中声明它接受或返回一个 `Guess` 类型的参数或返回值，而不需要在其主体中进行任何额外的检查。

## 摘要

Rust的错误处理功能旨在帮助你编写更健壮的代码。  
`panic!`宏表示你的程序处于无法处理的状态，它允许你让程序停止运行，而不是继续处理无效或错误的数值。`Result`枚举利用Rust的类型系统来表明某些操作可能会失败，但你的代码可以从中恢复。你可以使用`Result`来通知调用你的代码的代码，也需要处理可能出现的成功或失败情况。在适当的场景下使用`panic!`和`Result`，可以使你的代码在面临不可避免的问题时更加可靠。
