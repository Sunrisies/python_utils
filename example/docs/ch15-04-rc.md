## `Rc<T>`，引用计数智能指针

在大多数情况下，所有权是明确的：你清楚地知道哪个变量拥有某个值。然而，有时候一个值可能有多个所有者。例如，在图数据结构中，多条边可能指向同一个节点，而该节点在概念上属于所有指向它的边。除非一个节点没有任何指向它的边，因此没有所有者，否则不应该对这个节点进行清理。

你必须通过使用Rust类型`Rc<T>`来显式地启用多个所有权。⊂PH0⊘是_引用计数_的缩写。⊂PH1⊘类型会记录对某个值的引用次数，以确定该值是否仍然被使用。如果某个值没有任何引用，那么该值可以被清理，而不会使任何引用变得无效。

将 `Rc<T>` 想象成家庭客厅里的电视。当一个人进来看电视时，他们会打开电视。其他人也可以进入房间观看电视。当最后一个人离开房间时，他们会关闭电视，因为电视不再被使用了。如果有人在其他人还在观看电视时关闭电视，那么剩下的观众肯定会大吵大闹的！

当我们希望为程序的多个部分在堆上分配一些数据，并且在编译时无法确定哪个部分会最后使用这些数据时，我们会使用 `Rc<T>` 类型。如果我们能够知道哪个部分会最后使用这些数据，那么我们可以让那个部分成为数据的所有者，此时，编译时生效的正常所有权规则就会起作用。

请注意，`Rc<T>`这一语法仅适用于单线程场景。在第十六章中讨论并发时，我们会介绍如何在多线程程序中实现引用计数。

尝试使用我们定义的 `List` 和 `Box<T>` 来实现这一场景，但如清单 15-17 所示，这样做是行不通的。

<Listing number="15-17" file-name="src/main.rs" caption="Demonstrating that we’re not allowed to have two lists using `Box<T>` that try to share ownership of a third list">

```rust,ignore,does_not_compile
enum List {
    Cons(i32, Box<List>),
    Nil,
}

use crate::List::{Cons, Nil};

fn main() {
    let a = Cons(5, Box::new(Cons(10, Box::new(Nil))));
    let b = Cons(3, Box::new(a));
    let c = Cons(4, Box::new(a));
}

```

</Listing>

The `Cons` variants own the data they hold, so when we create the `b` list, `a`
is moved into `b` and `b` owns `a`. Then, when we try to use `a` again when
creating `c`, we’re not allowed to because `a` has been moved.

We could change the definition of `Cons` to hold references instead, but then
we would have to specify lifetime parameters. By specifying lifetime
parameters, we would be specifying that every element in the list will live at
least as long as the entire list. This is the case for the elements and lists
in Listing 15-17, but not in every scenario.

Instead, we’ll change our definition of `List` to use `Rc<T>` in place of
`Box<T>`, as shown in Listing 15-18. Each `Cons` variant will now hold a value
and an `Rc<T>` pointing to a `List`. When we create `b`, instead of taking
ownership of `a`, we’ll clone the `Rc<List>` that `a` is holding, thereby
increasing the number of references from one to two and letting `a` and `b`
share ownership of the data in that `Rc<List>`. We’ll also clone `a` when
creating `c`, increasing the number of references from two to three. Every time
we call `Rc::clone`, the reference count to the data within the `Rc<List>` will
increase, and the data won’t be cleaned up unless there are zero references to
it.

<Listing number="15-18" file-name="src/main.rs" caption="A definition of `List` that uses `Rc<T>`">

```rust
enum List {
    Cons(i32, Rc<List>),
    Nil,
}

use crate::List::{Cons, Nil};
use std::rc::Rc;

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    let b = Cons(3, Rc::clone(&a));
    let c = Cons(4, Rc::clone(&a));
}

```

</Listing>

我们需要添加一条 `use` 语句，以便将 `Rc<T>` 纳入范围，因为它不在前文中。在 `main` 中，我们创建了一个包含 `5` 和 `10` 的列表，并将其存储在一个新的 `Rc<List>` 中，该 `Rc<List>` 又位于 `a` 中。然后，当我们创建 `b` 和 `c` 时，我们调用 `Rc::clone` 函数，并将 `Rc<List>` 的引用作为参数传递给 `a`。

我们本可以将 `a.clone()` 作为替代，而不是 `Rc::clone(&a)`。不过，Rust 的惯例是在这种情况下使用 `Rc::clone`。`Rc::clone` 的实现并不会像大多数类型对 `clone` 的实现那样，对所有数据进行深度复制。对 `Rc::clone` 的调用只会增加引用计数，这个过程并不需要花费太多时间。而深度复制数据则可能需要很长时间。通过使用 `Rc::clone` 来进行引用计数，我们可以直观地区分出那些需要深度复制的克隆对象，以及那些只是增加引用计数的克隆对象。在查找代码中的性能问题时，我们只需要考虑那些需要深度复制的克隆对象，而可以忽略对 `Rc::clone` 的调用。

### 克隆操作以增加引用计数

让我们修改 Listing 15-18 中的示例，这样我们就可以看到在创建和删除对 `Rc<List>` 的引用时，reference 计数是如何变化的。

在 Listing 15-19 中，我们将修改 `main`，使其在 list 周围有一个内部作用域。然后，我们可以观察到当 `c` 离开作用域时，引用计数会发生怎样的变化。

<Listing number="15-19" file-name="src/main.rs" caption="Printing the reference count">

```rust
// --snip--

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    println!("count after creating a = {}", Rc::strong_count(&a));
    let b = Cons(3, Rc::clone(&a));
    println!("count after creating b = {}", Rc::strong_count(&a));
    {
        let c = Cons(4, Rc::clone(&a));
        println!("count after creating c = {}", Rc::strong_count(&a));
    }
    println!("count after c goes out of scope = {}", Rc::strong_count(&a));
}

```

</Listing>

在程序的每个引用计数发生变化的点，我们都会打印出引用计数的值。这个数值是通过调用 `Rc::strong_count` 函数得到的。这个函数被命名为 `strong_count`，而不是 `count`，因为 `Rc<T>` 类型也包含 `weak_count`；我们将在[“使用 `Weak<T>` 防止引用循环”][preventing-ref-cycles]<!-- ignore -->中了解 `weak_count` 的用途。

这段代码会输出以下内容：

```console
$ cargo run
   Compiling cons-list v0.1.0 (file:///projects/cons-list)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.45s
     Running `target/debug/cons-list`
count after creating a = 1
count after creating b = 2
count after creating c = 3
count after c goes out of scope = 2

```

我们可以看到，在 `a` 中，`Rc<List>` 的初始引用计数为 1；然后，每次我们调用 `clone` 时，引用计数会增加 1。当 `c` 超出作用域时，引用计数会减少 1。我们不需要像调用 `Rc::clone` 来增加引用计数那样，来调用函数来减少引用计数：`Drop`  trait 的实现会在 `Rc<T>` 的值超出作用域时，自动减少引用计数。

在这个例子中，我们无法看到的是，当 `b` 和 `a` 在 `main` 结束时退出作用域时，计数变为0，而 `Rc<List>` 则被完全清理。使用 `Rc<T>` 则允许一个值拥有多个所有者，而计数则确保该值在任何一个所有者仍然存在的情况下仍然有效。

通过不可变引用，`Rc<T>`允许你在程序的多个部分之间仅进行数据读取。如果`Rc<T>`也允许使用多个可变引用，那么可能会违反第4章中讨论的借用规则之一：对同一位置进行多次可变借用会导致数据竞争和不一致性。不过，能够修改数据是非常有用的！在下一节中，我们将讨论内部可变性模式，以及你可以与`Rc<T>`结合使用的`RefCell<T>`类型，以配合这种不可变性限制来工作。
