## 不安全Rust

到目前为止，我们所讨论的所有代码都遵循了Rust的内存安全保证，这些保证在编译时就会得到落实。不过，Rust内部还隐藏着另一种语言，它并不强制执行这些内存安全保证。这种语言被称为“不安全Rust”。虽然它的运作方式与普通Rust相同，但它为我们提供了额外的功能。

Unsafe Rust的存在是因为，从本质上讲，静态分析是一种保守的方法。当编译器试图判断代码是否遵守了相关规范时，与其接受一些无效的代码，不如拒绝一些合法的代码。虽然代码本身可能没有问题，但如果Rust编译器没有足够的信息来做出判断，它就会拒绝该代码。在这种情况下，你可以使用`unsafe`代码来告诉编译器：“相信我，我知道自己在做什么。”不过需要注意的是，使用`unsafe`代码需要你自己承担风险：如果你错误地使用了这种代码，可能会因为内存安全问题而出现问题，比如空指针解引用。

Rust之所以有“不安全”的版本，另一个原因是底层计算机硬件本身具有不安全性。如果Rust不允许执行不安全操作，那么许多任务就无法完成。Rust需要允许开发者进行低级系统编程，比如直接与操作系统交互，甚至编写自己的操作系统。进行低级系统编程正是该语言的目标之一。让我们来探讨一下如何使用Rust的“不安全”版本来完成这些任务。

### 行使不安全的超能力

要切换到不安全的Rust环境，可以使用`unsafe`这个关键字，然后开始一个新的代码块，其中包含了不安全的代码。在不安全的Rust环境中，你可以执行一些在安全Rust中无法进行的操作，这些操作我们称之为“不安全的超能力”。这些超能力包括：

1. 解引用原始指针。  
1. 调用不安全的函数或方法。  
1. 访问或修改可变的静态变量。  
1. 实现不安全的特性。  
1. 访问 `union`s 的字段。

需要明白的是，`unsafe`并不会关闭借用检查器，也不会禁用Rust的其他安全检查机制：如果你在unsafe代码中使用了引用，这些引用仍然会被进行检查。`unsafe`这个关键字只是让你能够访问这五个特性，而这些特性不会由编译器进行内存安全性的检查。尽管如此，你在unsafe代码块内仍然可以获得一定程度的安全性保障。

此外，`unsafe`并不意味着该代码块中的内容一定具有危险性，或者一定会存在内存安全方面的问题。其目的是让程序员能够确保位于`unsafe`代码块中的代码能够以合法的方式访问内存。

人们都会犯错，错误是难免的，但是通过要求这五种不安全操作必须位于带有 ``unsafe`` 注释的块中，你就可以确保与内存安全相关的任何错误都发生在 ``unsafe`` 块内。尽量保持 ``unsafe`` 块的小规模；这样在排查内存漏洞时你会感到庆幸。

为了尽可能隔离不安全的代码，最好将这些代码封装在一个安全的抽象层中，并提供安全的API。我们将在后面的章节中讨论如何设计不安全的函数和方法。标准库中的某些部分就是通过将不安全的代码进行安全化处理来实现的。将不安全的代码封装在安全的抽象层中，可以防止``unsafe``被泄露到那些你可能想要使用``unsafe``实现的功能的地方，因为使用安全的抽象层本身就是一种安全的做法。

让我们依次来看看这五个“不安全超级功能”。同时，我们还会了解一下一些能够为不安全代码提供安全接口的抽象概念。

### 对原始指针进行解引用

在第四章的[“悬空引用”][dangling-references]<!-- ignore
-->部分中，我们提到编译器会确保引用始终有效。Unsafe Rust中有两种新的类型，称为_原始指针_，它们与引用类似。与引用一样，原始指针可以是不可变的，也可以是可变的，并且分别被表示为`*const T`和`*mut T`。星号并不是解引用运算符；它是类型名称的一部分。在原始指针的上下文中，_不可变_意味着解引用后的指针不能直接被重新赋值。

与引用和智能指针不同，原始指针：

- 允许通过同时使用不可变指针和可变指针，或者多个可变指针指向同一个位置来忽略借用规则  
- 不保证会指向有效的内存地址  
- 允许值为 null  
- 不实现任何自动清理机制

通过选择不让Rust强制执行这些保证，你可以放弃有保障的安全性，以换取更高的性能，或者能够与那些Rust的保证不适用的其他语言或硬件进行交互。

#### 使用其他语言调用Rust函数

我们还可以使用 ``extern`` 来创建一个接口，使得其他语言能够调用Rust函数。无需创建完整的 ``extern`` 块，我们可以在 ``fn`` 关键字之前添加 ``extern`` 关键字，并指定要使用的ABI。同时，我们还需要添加 ``#[unsafe(no_mangle)]`` 注释，以告知Rust编译器不要修改该函数的名称。所谓“命名混淆”，是指编译器将我们给函数的名称改为一个包含更多信息的名称，但这个名称对其他编译过程中的部分来说更有用，但可读性较差。每种编程语言编译器对名称的混淆方式都略有不同，因此为了让Rust函数能够被其他语言引用，我们必须禁用Rust编译器的名称混淆功能。这样做是不安全的，因为在没有内置混淆功能的情况下，不同库之间可能会出现名称冲突。因此，我们有责任确保所选的名称在未经混淆的情况下是安全可导出的。

在下面的示例中，我们使得 `call_from_c` 这个函数能够在 C 代码中访问，前提是该函数已经被编译成共享库，并且从 C 代码中进行了链接。

```
#[unsafe(no_mangle)]
pub extern "C" fn call_from_c() {
    println!("Just called a Rust function from C!");
}
```

这种对`extern`的使用要求`unsafe`仅存在于属性中，而不应出现在`extern`块上。

常数与不可变静态变量之间的一个微妙区别在于：静态变量中的值在内存中具有固定的地址。使用这些值时，总是可以访问相同的数据。而常数则可以在被使用时复制其数据。另一个区别是，静态变量可以是可变的。访问和修改可变的静态变量是_不安全的_。清单20-11展示了如何声明、访问和修改一个名为`COUNTER`的可变静态变量。

与常规变量一样，我们使用 ``mut`` 关键字来指定可变性。任何从 ``COUNTER`` 读取或写入数据的代码都必须位于 ``unsafe`` 块内。正如预期的那样，清单 20-11 中的代码会编译并打印出 ``COUNTER: 3``，因为该代码是单线程执行的。如果多个线程访问 ``COUNTER``，可能会导致数据竞争，因此这属于未定义行为。因此，我们需要将整个函数标记为 ``unsafe``，并说明安全限制，以便调用该函数的人了解允许和禁止做的事情。

每当我们编写不安全函数时，通常会以 `SAFETY` 开头的注释来说明调用者需要做什么以确保函数能够安全地被调用。同样地，每当我们进行不安全操作时，也会以 `SAFETY` 开头的注释来解释如何遵守安全规则。

此外，编译器默认会拒绝任何试图通过编译时的检查来创建对可变静态变量的引用的行为。您必须明确选择放弃这种检查的保护机制，方法是添加 ``#[allow(static_mut_refs)]`` 注解，或者通过一些原始借用运算符来创建原始指针来访问可变静态变量。这包括那些以不可见方式创建引用的情况，比如在 ``println!`` 中使用的引用。要求对静态可变变量的引用必须通过原始指针来创建，这有助于使使用这些变量的安全要求更加明确。

由于可变数据可以全局访问，因此很难确保不会出现数据竞争问题。这就是为什么Rust认为可变的静态变量是不安全的。在可能的情况下，最好使用我们在第16章中讨论的并发技术和线程安全智能指针，这样编译器就能确保来自不同线程的数据访问是安全的。

### 实现不安全特质

我们可以使用 ``unsafe`` 来实现一个不安全的特性。当一个特性的至少其中一个方法包含编译器无法验证的不变性时，该特性就被称为不安全的。我们通过在 ``trait`` 之前添加 ``unsafe`` 关键字来声明一个特性是 `unsafe`，并且还将该特性的实现标记为 `unsafe`，如清单 20-12 所示。

通过使用 ``unsafe impl``，我们承诺会遵守那些编译器无法验证的不变量。

例如，回想一下我们在第16章的[“Extensible Concurrency with `Send` and `Sync`”][send-and-sync]<!-- ignore -->这一节中讨论的`Send`和`Sync`标记特性。如果我们所定义的类型完全由其他实现了`Send`和`Sync`的类型组成，那么编译器会自动实现这些特性。但是，如果我们定义的一种类型中包含一种未实现`Send`或`Sync`的类型，比如原始指针，并且我们希望将该类型标记为`Send`或`Sync`的话，我们就必须使用`unsafe`来实现这一点。Rust无法验证我们的类型是否能够安全地跨线程传输或从多个线程处访问；因此，我们需要手动进行这些检查，并通过`unsafe`来标明这一点。

### 访问联合类型的字段

最后一个适用于`unsafe`的操作是访问联合体的字段。  
“联合体”类似于`struct`，但一个联合体在某一时刻只能使用一个已声明的字段。联合体主要用于与C代码中的联合体进行交互。由于Rust无法保证联合体实例中存储的数据类型，因此访问联合体字段是不安全的。您可以在[Rust参考手册][unions]中了解更多关于联合体的信息。

### 使用 Miri 检查不安全代码

在编写不安全代码时，你可能希望确认所写的代码确实安全且正确。实现这一目标的最佳方法之一是使用Miri，这是Rust官方提供的一个用于检测未定义行为的工具。虽然借用检查器是一种在编译时生效的静态工具，但Miri则是一种在运行时生效的动态工具。它通过运行你的程序或其测试套件来检查代码，并识别你是否违反了Rust规范所规定的规则。

使用Miri需要安装最新版本的Rust（我们在[附录G：Rust的制造过程与“Nightly Rust”](http://nightly)中有更多介绍）。您可以通过运行`rustup +nightly component add miri`来安装最新版本的Rust以及Miri工具。这一操作并不会改变项目所使用的Rust版本；它只是将Miri工具添加到您的系统中，以便您可以在需要时使用它。您可以通过输入`cargo +nightly miri run`或`cargo +nightly miri test`来在项目中运行Miri。

Miri并不能捕捉到你在编写不安全代码时可能犯的所有错误。Miri是一个动态分析工具，因此它只能发现实际运行时的代码问题。这意味着你需要结合良好的测试技术来使用它，以提高对编写的不安全代码的信心。此外，Miri并不能涵盖代码中所有可能存在的缺陷。

换句话说：如果Miri确实发现了问题，那么显然存在错误；但仅仅因为Miri没有发现错误，并不意味着就没有问题。不过，Miri确实有可能发现很多问题。试着在本章中其他不安全代码的例子上运行它，看看它会显示什么结果！

### 正确使用不安全代码

使用 ``unsafe`` 来使用刚才提到的五种超能力之一并没有错，也不会受到指责。不过，要确保 ``unsafe`` 代码的正确性则更为复杂，因为编译器无法自动保证内存安全。当你有理由使用 ``unsafe`` 代码时，你可以这样做。而拥有明确的 ``unsafe`` 注解则有助于在问题出现时更容易追踪问题的来源。每当你编写不安全代码时，可以使用 Miri 来帮助你更确信自己所写的代码符合 Rust 的规则。

如果您想更深入地了解如何在 unsafe Rust 环境中有效工作，请阅读 Rust 官方指南中关于 `unsafe` 的内容，以及《Rustonomicon》[nomicon]。
