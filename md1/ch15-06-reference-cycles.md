## 引用循环可能会导致内存泄漏

Rust的内存安全特性虽然能有效防止内存泄漏，但这种情况仍有可能发生。完全避免内存泄漏并非Rust所保证的功能，因此内存泄漏在Rust中仍然是可预防的。通过使用`Rc<T>`和`RefCell<T>`，Rust确实允许内存泄漏的发生。在这种情况下，某些引用可能会形成一个循环引用，从而导致内存泄漏。因为循环中的各个对象的引用计数永远不会降为零，这些对象的值也就永远不会被释放。

创建引用循环并不容易实现，但也不是不可能。如果你有包含 ``Rc<T>`` 值的 ``RefCell<T>`` 值，或者存在类似的嵌套类型组合，并且这些类型具有可变性以及引用计数机制，那么你必须确保不会创建引用循环；你不能指望 Rust 能够自动检测这些循环。在程序中创建引用循环属于逻辑错误，你应该通过自动化测试、代码审查以及其他软件开发实践来减少这种错误的发生。

### 使用 `Weak<T>` 防止引用循环

到目前为止，我们已经证明了调用 ``Rc::clone`` 会增加 ``Rc<T>`` 实例的 ``strong_count``，而 ``Rc<T>`` 实例只有在其 ``strong_count`` 为 0 时才会被清理。你还可以通过调用 ``Rc::downgrade`` 并传递对 ``Rc<T>`` 的引用，来在 ``Rc<T>`` 实例中创建对该值的弱引用。*强引用*是共享 ``Rc<T>`` 实例所有权的方式。*弱引用*则不会表达所有权关系，它们的数量也不会影响 ``Rc<T>`` 实例的清理时机。它们不会导致引用循环，因为任何涉及弱引用的循环都会在相关值的强引用计数变为 0 时被打破。

当你调用 ``Rc::downgrade`` 时，你会得到一个类型为 ``Weak<T>`` 的智能指针。  
与在 ``Rc<T>`` 实例中增加 ``strong_count`` 的值1不同，调用 ``Rc::downgrade`` 会令 ``weak_count`` 的值增加1。类型 ``Rc<T>`` 使用 ``weak_count`` 来记录有多少个引用指向 `__INLINE_CODE_97`，这与 ``strong_count`` 的机制类似。不同之处在于，``weak_count`` 的值不需要为0，这样才能让 ``Rc<T>`` 实例得到清理。

因为 `Weak<T>` 所引用的数值可能被删除了，所以要对由 `Weak<T>` 指向的数值进行操作，必须确保该数值仍然存在。可以通过调用 `upgrade` 方法来实现这一点，该方法会返回一个 `Option<Rc<T>>` 类型的值。如果 `Rc<T>` 所引用的数值尚未被删除，那么结果将是 `Some`；如果 `Rc<T>` 所引用的数值已被删除，那么结果将是 `None`。由于 `upgrade` 返回的是 `Option<Rc<T>>` 类型的值，因此 Rust 会确保 `Some` 和 `None` 这两种情况都能得到正确处理，从而不会出现无效的指针引用。

例如，我们不会使用一种只有当前项与下一个项之间关系的列表，而是会创建一个树形结构，其中每个项不仅了解其子项，还了解其父项。

在 `leaf` 被创建之后，其 `Rc<Node>` 的强计数变为 1，而弱计数则变为 0。在内部作用域中，我们创建了 `branch` 并将其与 `leaf` 关联起来。此时，当我们打印这些计数时，`branch` 中的 `Rc<Node>` 将具有强计数 1 和弱计数 1（因为 `leaf.parent` 指向了 `branch`，而 `branch` 又通过 `Weak<Node>` 引用了 `Rc<Node>`）。当我们在 `leaf` 中打印这些计数时，会发现其强计数变为 2，因为 `branch` 现在在 `branch.children` 中保存了对 `leaf` 的 `Rc<Node>` 的克隆，不过其弱计数仍然为 0。

## 总结

本章介绍了如何使用智能指针来做出与Rust默认使用常规引用时不同的保证和权衡。`Box<T>`类型具有已知的大小，并且指向堆上分配的数据。`Rc<T>`类型会跟踪对堆上数据的引用次数，这样数据就可以有多个所有者。`RefCell<T>`类型具有内部可变性，这种类型可以在我们需要不可变类型但需要修改该类型的内部值时使用；它还在运行时而不是编译时强制执行借用规则。

此外，还讨论了`Deref`和`Drop`这两个特性，它们实现了智能指针的许多功能。我们探讨了可能导致内存泄漏的引用循环问题，以及如何使用`Weak<T>`来防止这种情况的发生。
